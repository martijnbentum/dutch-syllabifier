# Design note: sonority-based onset rule as fallback / whitelist check

Status: **mode 2 (lint) implemented; mode 1 (fallback) not.** The rule lives
in `phonotactics.is_sonority_legal_onset` and the lint in
`tests/test_phonotactics.py`: whitelisted onsets the rule rejects fail hard;
SSP-valid onsets attested in CELEX but missing from `legal_onsets.json` are
reported as an informational pytest skip. Onset legality for boundary
placement remains a pure whitelist lookup against `legal_onsets.json`.
This note sketches the rule-based companion based on the Sonority Sequencing
Principle (SSP). Reference: Booij (1995), *The Phonology of Dutch*.

## The principle

Within a syllable, sonority rises from the onset edge toward the nucleus and
falls toward the coda edge. An onset cluster is well-formed when each segment is
strictly more sonorous than the one before it (moving toward the nucleus), by at
least some minimum distance.

## A Dutch sonority scale (high -> low)

```
vowels                         (nucleus, not part of onset)
glides        j  ʋ             6
liquids       l  r             5
nasals        m  n  ŋ          4
voiced fric   v  z  ɣ          3
voiceless fric f  s  x  h      2
voiced stop   b  d  ɡ          1
voiceless stop p  t  k         0
```

(Exact tier values are tunable; what matters is the ordering and a configurable
minimum sonority distance between adjacent onset segments.)

## The one principled exception: /s/-appendix

Dutch onset clusters that begin with `/s/` violate the SSP: `sp st sk sf sx`
have *falling* then *rising* sonority, and `spr str skr sxr spl` add a liquid.
These are not generated by the rule; they are sanctioned as an `s`-appendix that
sits outside the normal sonority profile. Any rule-based check must whitelist
the leading `/s/` (and only `/s/`) as an appendix, then apply the SSP to the
remainder.

## Proposed use

Two non-exclusive modes:

1. **Fallback acceptance.** When a cluster is absent from `legal_onsets.json`,
   accept it if it is SSP-valid (after stripping a leading `/s/` appendix) and
   within the Dutch max onset size (3). This makes the whitelist a fast path and
   the rule a safety net for gaps (e.g. a missing but legal `Cr`/`Cl`).

2. **Whitelist lint.** Offline, compare the rule against the whitelist:
   - flag any *whitelisted* onset that the rule rejects and that is not a
     sanctioned `s`-appendix (candidate data error);
   - flag any *SSP-valid* cluster the whitelist omits (candidate gap, e.g. the
     recently-dropped `spl`), for human review — not auto-added.

## Sketch

```python
SONORITY = {...}              # phone -> int, per the scale above
MIN_DISTANCE = 1              # tunable; Dutch onsets want a clear rise
MAX_ONSET = 3

def sonority_legal_onset(labels):
    '''True if labels form an SSP-valid Dutch onset (s-appendix allowed).'''
    seq = list(labels)
    if len(seq) > MAX_ONSET:
        return False
    if len(seq) >= 2 and seq[0] == 's':   # strip the s-appendix, check the rest
        seq = seq[1:]
    for a, b in zip(seq, seq[1:]):
        if SONORITY[b] - SONORITY[a] < MIN_DISTANCE:
            return False
    return True
```

Caveats / Dutch specifics to encode before trusting the rule:
- `/tl/ /dl/` are SSP-valid but not native Dutch onsets -> need an extra
  coronal-stop + `/l/` ban, or keep the whitelist authoritative and use the rule
  only as a lint.
- glide-final onsets `Cʋ` (`tw dw kw zw sw`) are fine under SSP.
- `/ŋ/` never an onset (handled by it never appearing onset-initially).

Recommended first step: ship mode 2 (lint) only, so the whitelist stays the
source of truth while the rule surfaces drift and gaps.
